Why the current design is reasonable

ChoiceReason
Local-first PWABest fit for a child-facing practice app: no backend required, works offline, fast, private by default, installable on iPad/phone/desktop.
Dexie/IndexedDBStructured local storage with transactions, indexes, and schema migration support.
Event log as source of truthAllows rebuilding derived mastery/scheduler caches and makes cross-device sync safer.
FSRS via adapterGood separation: app logic uses MathFan review grades; only one file knows the third-party FSRS API.
Deterministic item IDsCritical for generated questions. Same ID can be reconstructed after reload/sync without storing every generated item forever.
Drive appDataFolder syncLower operational burden than running a server; data stays in the user’s Google account.
Optional Gemini tutorPreserves core offline/local practice while giving families an opt-in hint system.

Current tradeoffs

TradeoffImpact
Manual screen state instead of URL routingSimpler for a small app, but harder to deep-link, restore exact sub-screen state, or use browser back/forward.
Inline styles in many componentsFast iteration, but harder to enforce theme consistency and responsive design at scale.
Snapshot sync instead of row-level backend syncSimple and private, but larger snapshots and coarser conflict handling as data grows.
API key in localStorageConvenient for personal/family use, but not ideal for public app security.
Multiple profiles partially implementedDB supports it, but startup/profile switching UX needs a real chooser.

Things to take care of during future changes

Do not breakCanonical event log

New stats/mastery should prefer mathAnswerEvents. Keep attempts as compatibility unless intentionally migrating.

Do not breakFirst-attempt scheduling

Retries must stay logged but must not update FSRS item state.

Be carefulGenerated item IDs

Every generated item type needs a stable ID and a matching makeItemFromId parser.

Be carefulSync merge

After merging remote snapshot, derived caches must be rebuilt from merged events.

Be carefulGoogle sign-in timing

GIS token requests are sensitive to user gestures, popups, local origins, and COOP behavior.

TestSpeech auto-advance

Advance only after answer speech finishes, or the next prompt may cut off feedback audio.

TestService worker caching

When UI looks stale, test with version/commit display, fresh query params, and SW unregister/reload.

Architecture ruleFSRS import boundary

Keep ts-fsrs isolated in fsrsAdapter.ts; callers should use applyReview/applyFsrsReview.

Scalability assessment

DimensionCurrent stateScaling concern
More grades/skillsFeature folders and deterministic item factories scale reasonably.Skill mapping may become brittle if rules remain hard-coded; consider data-driven curriculum definitions.
More students/devicesIndexedDB + Drive snapshot works for family use.Large snapshots and merge time can grow; row-level sync or chunked event sync may be needed.
More visualsVisualModel abstraction supports adding types.Bundle size and mobile performance need monitoring; lazy-load heavy visuals if needed.
More AI useOptional BYO Gemini key is simple.Public deployment needs server-side key protection, abuse control, and safety logging/privacy review.
More analyticsEvent log supports recomputation.IndexedDB queries over many events may need indexes, materialized summaries, or paginated processing.
Classroom useCurrent design is family/local-first.Needs account model, roster, admin dashboard, consent/privacy workflows, and backend sync.

Better design options by stage

Near-term: keep local-first

Add profile picker, export/import JSON, stronger version/build display, more regression tests, and data repair tools using event rebuild.

Medium-term: data-driven curriculum

Move skill metadata, prerequisite advisory, item generators, and mappings into declarative configs with validation tests.

Medium-term: router adoption

Use React Router for shareable URLs and back-button behavior while keeping the app shell simple.

Public-scale: backend sync

Move from Drive snapshots to Firestore/Supabase/custom backend if classroom/large multi-device sync becomes a goal.

Public-scale: AI proxy

Replace local API keys with a sign-in-gated serverless proxy, rate limits, and model policy controls.

Priority improvements

  1. Profile picker: DB supports multiple profiles, but startup selection and “switch student” need a real choose-existing flow.
  2. Export/import: Use the existing snapshot format as the starting point, but add validation and “new learning track” semantics.
  3. Curriculum config: Centralize skill mapping and item generation coverage tests.
  4. Sync/privacy: Add optional encryption or at least clear warnings for Drive snapshots.
  5. Router: Adopt URL routes if deep links/history become important.
